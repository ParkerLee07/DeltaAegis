# DeltaAegis v0.45 — Telemetry Trust Runtime

## Scope

This checkpoint implements the approved telemetry-trust checkpoints 1–5 in
one guarded installation:

1. deterministic quality decision runtime;
2. durable automated-decision and review ledger;
3. ingestion-effect enforcement and replayable current-state projection;
4. authenticated review and policy-permitted override backend;
5. API, report, asset-detail, operator, and Telemetry Quality Center visibility.

It does not change NetSniper taxonomy, evidence weights, scan behavior, or
classification calibration.

## Four quality states

- `ACCEPTED`: normal positive analysis is allowed. Absence effects remain
  independently gated by proven coverage.
- `DEGRADED`: positive assets, services, findings, and strictly stronger
  classification evidence may be applied. Absence mutation, alert resolution,
  and high-severity classification escalation are blocked.
- `QUARANTINED`: source evidence is retained for review but does not mutate
  operational current state.
- `REJECTED`: only the decision receipt and audit metadata are retained.

Automated state is immutable. Current reviewed state is stored separately. The
public evaluator emits the exact existing
`deltaaegis-telemetry-quality-decision-v1` schema; internal ledger identifiers
and retained paths are exposed separately by detail APIs.

## Lazy compatibility-preserving initialization

Normal `connect()` preserves the characterized v0.44 startup table inventory.
`install.sh` syntax-checks both v0.45 core modules whenever they are present.
The root facade retains a narrow import fallback only for characterized legacy
lifecycle fixtures that intentionally copy the pre-v0.45 minimal source set;
telemetry-trust feature use still requires the installed modules.
The v0.45 quality ledger and current-state projection are created transactionally
only when ingestion, the Telemetry Quality Center, review/override, report
disclosure, or projected-state access first requires them. The first projection
operation seeds the latest legacy accepted snapshot per scope without inventing
historical v0.45 decisions.

## Durable evidence and current state

Trusted and quarantined source bundles are copied atomically into:

```text
~/DeltaAegis/telemetry-evidence/
├── trusted/
└── quarantine/
```

The original NetSniper bundle is never rewritten. Source and retained copies
are hashed before trust; an existing evidence-store destination is reused only
when its full bundle digest matches the immutable decision ledger.

Historical snapshot and observation tables remain the immutable evidence
ledger. A run identifier is also content-bound across both the new quality
ledger and pre-v0.45 snapshot history. Reuse with different content—or reuse
of a legacy run identifier whose original content can no longer be verified—
fails closed as `run_id_hash_conflict`.

Operational state is projected into:

- `telemetry_current_assets`
- `telemetry_current_services`
- `telemetry_current_findings`

Overrides rebuild the affected network scope and lifecycle state from the
latest legacy accepted seed plus retained v0.45 decisions instead of mutating
historical observations or inventing retroactive event records.

## Degraded evidence rules

A degraded run:

- can add or refresh directly observed assets, services, and findings;
- cannot remove an unobserved asset;
- cannot close an unobserved service;
- cannot resolve an alert because evidence disappeared;
- cannot independently raise risk above score 64 (`MEDIUM`);
- cannot replace classification unless the new decision rank, confidence,
  evidence support, contradiction posture, and validation state are strictly
  stronger and the semantic fingerprint is distinct;
- cannot use `review_only`, `display_only`, `no_action`, or equivalent
  operator dispositions as a replacement classification.

Accepted evidence independently supporting a higher risk is not reduced by the
degraded-only ceiling. Asset, service, and finding provenance is tracked per
record, so a new degraded service or finding never inherits accepted status
merely because the parent asset was previously accepted.

## Review and override

Permissions:

- `VIEWER`: read quality information only;
- `ANALYST`: add review annotations;
- `ADMIN`: add annotations and apply policy-permitted overrides.

Review and override require an authenticated dashboard session. Legacy
dashboard tokens and API tokens cannot perform these actions. Caller-supplied
actor identity fields are rejected.

Permitted transitions are defined by
`contracts/v0.45/telemetry-quality-policy.json`. `REJECTED` is
non-overridable. Every review and override requires a reason and creates an
append-only review record plus an access-audit event.

## Operator surfaces

- `/operator/telemetry-quality`
- `GET /api/telemetry-quality`
- `GET /api/telemetry-quality/detail`
- `POST /api/telemetry-quality/review`
- `POST /api/telemetry-quality/override`

Generated investigation reports include a Telemetry Quality section.
Asset-detail and current-state payloads include quality provenance without
placing the full NetSniper reasoning object in asset-list payloads.

## Compatibility

All manual, dashboard, scheduled, worker, and completion-recovery ingestion
paths continue to converge on `ingest_manifest()`. The centralized function
now performs quality evaluation, retention, durable decision storage, and
effect enforcement before committing.

Legacy databases are seeded conservatively from the latest accepted snapshot
per network scope. The migration does not invent historical v0.45 decisions.

## Intentional v0.44 Stage 4 transition

The repository-owned emergency demo uses the legacy `netsniper-run-v2`
contract. DeltaAegis v0.44 characterized those bundles as `ACCEPTED` and the
second demo receipt included eleven events, one of which was an
absence-derived `NETSNIPER_FINDING_REMOVED` event.

The approved v0.45 policy intentionally changes that behavior. Legacy v2
bundles default to `DEGRADED`; the demo emits ten positive service/finding
addition events, retains the earlier finding in current state, and does not
emit the removal event. `tools/validate_v0_45_v0_44_ingest_transition.py`
reuses the unchanged v0.44 ownership, normalization, trust-boundary, scope,
and identity checks while validating this explicit policy transition.

The v0.44 Stage 8 route fingerprint is also intentionally extended by the
Quality Center. `tools/validate_v0_45_v0_44_web_transition.py` proves that all
57 characterized routes remain unchanged and that the only API additions are
the approved quality list, detail, review, and override endpoints. The
protected `/operator/telemetry-quality` page is validated separately because
the historical route fingerprint did not include that path shape.

## Applicable predecessor compatibility gate

The frozen v0.44 Stage 4 and Stage 8 validators cannot pass unchanged because
they assert that legacy v2 telemetry remains accepted and that no route may be
added. Both assertions directly conflict with the approved v0.45 behavior.
They are not edited or bypassed. Instead,
`tools/validate_v0_45_v0_44_compatibility_all.sh` runs the unchanged v0.44
component validators and predecessor suites once, substituting only the two
explicit transition validators for those incompatible frozen assertions.

## Stage 0G quality-policy correction

The runtime continues to consume the unchanged NetSniper v2.1 capability and
host-classification contracts. Collector completion, failure, unavailability,
inventory preservation, and negative-evidence eligibility are authoritative for
bundle sufficiency; root versus non-root execution is not independently a
downgrade when requested collectors completed.

Host classifications marked `review` or `unknown` remain visible and
confidence-gated without downgrading unrelated complete telemetry.
Identity collisions remain quarantining. Persisted Stage 0F decisions are not
rewritten; new decisions are recorded under `deltaaegis-v0.45-stage0g`.

## Deep audit hardening

The final v0.45 candidate validates every retained run identifier as a bounded,
path-free value and confines trusted/quarantine evidence destinations beneath
the configured evidence root. Schema helpers no longer use SQLite
`executescript()` inside operational paths, so caller-owned transactions are
not committed implicitly.

Ingestion and override workflows use explicit database transactions and
compensating evidence operations. A failed ingest removes evidence created by
that attempt. An override stages and verifies its target copy, commits the
ledger, snapshot, projection, intelligence, and audit changes together, and
only then removes the superseded source copy.

NetSniper v2.1 acceptance now requires the complete capability contract:
explicit execution status, required completed collectors, integrity-bound
artifacts, consistent run and version identifiers, inventory/host-row
agreement, and unique nonempty host identifiers. A complete zero-host scan is
valid negative evidence when those independent coverage checks pass.

The dashboard escapes both quote characters in HTML attribute contexts.
Current-state risk merging keys assets by network scope and asset identity,
applies degraded-only ceilings to preexisting rows, and keeps independently
accepted risk support. Historical snapshots are retained as history without
replacing a newer current-state projection or emitting reverse deltas.

`tools/validate_v0_45_deep_bug_fixes.py` exercises these boundaries in addition
to the focused telemetry-trust and predecessor compatibility gates.
